# Kawang 后端结构与功能说明

## 1. 项目定位

`backend/kah` 是整个 AI 会员商城系统的后端服务，职责是：

- 对前台商城提供商品展示、公告展示、游客下单、游客查单、会员注册登录、会员订单查询接口。
- 对后台管理端提供管理员登录、商品管理、账号池管理、订单管理、公告管理接口。
- 负责真实的数据库读写，包括商品、订单、账号池、公告、管理员、会员等核心业务数据。
- 在“模拟下单”前提下，仍然真实完成库存校验、账号分配、订单入库和销量回写。

简单理解，这个后端不是只返回假数据的演示接口，而是一个已经具备基础业务闭环的商城服务。

## 2. 技术栈

当前后端使用的主要技术如下：

### 2.1 基础框架

- `Spring Boot 4.0.5`
- `Java 17`

Spring Boot 负责整个应用启动、依赖注入、Web 接口、配置绑定、安全链路和事务管理。

### 2.2 Web 与参数校验

- `spring-boot-starter-web`
- `spring-boot-starter-validation`

用途：

- 提供 REST API。
- 使用 `@RestController`、`@RequestMapping`、`@GetMapping`、`@PostMapping` 等注解定义接口。
- 使用 `jakarta.validation` 做参数校验，例如必填、长度、数量范围等。

### 2.3 安全与鉴权

- `spring-boot-starter-security`
- 自定义 JWT
- 自定义限流过滤器

用途：

- 区分公开接口、会员接口、管理员接口。
- 使用 JWT 实现无状态登录。
- 使用限流过滤器限制注册、登录、下单等高风险接口的频率。

### 2.4 数据访问

- `MyBatis`
- `MySQL`
- `spring-boot-starter-data-jdbc`（为数据库基础能力提供支持）

为什么这里选择 MyBatis：

- SQL 可控，适合订单、库存、账号池这类对细节要求高的场景。
- 可以清楚看到 `FOR UPDATE`、分页 SQL、动态 SQL。
- 对库存锁定、账号分配、统计回刷等逻辑更直观。

### 2.5 常用工具库

- `Lombok`
- `Jackson`

用途：

- Lombok 用来减少实体类和通用模型类中的样板代码。
- Jackson 用来处理 JSON 序列化，以及 `LocalDateTime` 这类时间字段输出。

## 4. 分层设计与职责

这个后端是一个比较标准的分层架构。

### 4.1 Controller 层

职责非常明确：

- 接收 HTTP 请求。
- 做基础参数绑定与校验。
- 调用 Service 接口。
- 返回统一 `ApiResponse`。

它不直接写业务 SQL，也不直接处理复杂事务。

### 4.2 Service 接口层

`service` 根包只定义“能力契约”，例如：

- `ProductFacadeService`
- `OrderFacadeService`
- `MemberUserService`
- `AdminProductService`
- `AdminAccountService`
- `AdminOrderService`
- `AdminAuthService`
- `NoticeService`

这样做的好处：

- 便于阅读职责。
- controller 依赖接口，不依赖具体实现。
- 后续如果要换实现、做测试替身或拆分模块，成本更低。

### 4.3 Service 实现层

实现都放在 `service.impl`。

例如：

- `ProductFacadeServiceImpl`
- `OrderFacadeServiceImpl`
- `MemberUserServiceImpl`
- `AdminProductServiceImpl`
- `AdminAccountServiceImpl`
- `AdminOrderServiceImpl`
- `AdminAuthServiceImpl`
- `NoticeServiceImpl`

这里是真正的业务核心层，负责：

- 业务校验。
- 事务控制。
- 调用 mapper。
- 对实体和 DTO 做映射。

### 4.4 Service 抽象基类

位于：

- `service.impl.base\AbstractServiceSupport`
- `service.impl.base\AbstractCrudService`

作用：

- 放公共断言逻辑。
- 放分页归一逻辑。
- 放通用 `requireById` 逻辑。
- 减少重复代码。

这次重构后，service 层形成了“接口 + impl + base”的结构，整体比以前直接把所有实现堆在一个包里更清晰。

### 4.5 Mapper 层

Mapper 层直接负责数据库访问。

特点：

- SQL 明确。
- 关键场景可以直接使用 `FOR UPDATE`。
- 订单列表和查单场景使用动态 SQL。
- 不通过 JPA 自动推断，业务更可控。

## 7. 核心业务流程

### 7.1 前台商品展示

对应服务：

- `ProductFacadeService`
- `ProductFacadeServiceImpl`

流程：

1. controller 接收商品列表请求。
2. service 调用 `ProductMapper.findActiveProducts()`。
3. 只返回状态为 `ACTIVE` 的商品。
4. service 把实体映射成前台 `ProductView`。
5. controller 用统一 `ApiResponse` 返回。

### 7.2 下单流程

对应核心服务：

- `OrderFacadeService`
- `OrderFacadeServiceImpl`

这是整个系统最关键的业务流程之一。

主流程如下：

1. 根据 `productId` 锁定商品记录。
2. 校验商品是否存在、是否上架。
3. 校验库存是否足够。
4. 从账号池里锁定足量可用账号。
5. 创建订单主记录。
6. 把账号逐个分配到当前订单。
7. 写入订单账号关系快照。
8. 回刷商品库存和销量统计。
9. 返回下单成功结果。

这里的关键点有两个：

- 商品和账号池都做了加锁，避免并发超卖。
- 虽然是“模拟下单”，但订单、账号分配、库存变化都是真实落库的。

### 7.3 游客查单

对应：

- `PublicOrderController`
- `OrderFacadeServiceImpl.queryByContact`
- `ShopOrderSqlProvider.buildContactQuerySql`

查单方式：

- 按 `buyer_contact` 查询
- 可选附加 `order_no`

返回结果只展示订单信息，不展示账号明文。

### 7.4 会员注册与登录

对应：

- `MemberUserService`
- `MemberUserServiceImpl`

注册流程：

1. 校验用户名是否已存在。
2. 使用 BCrypt 哈希密码。
3. 写入会员表。
4. 直接返回登录态 JWT。

登录流程：

1. 根据用户名查询会员。
2. 校验密码哈希。
3. 校验账号状态。
4. 更新最后登录时间。
5. 返回 JWT 和会员资料。

### 7.5 会员订单

会员下单时如果当前已登录：

- `shop_order.user_id` 会自动写入当前会员 ID。

这样会员后续访问：

- `GET /api/auth/orders`

就能看到自己绑定过的订单，而不需要每次都只靠联系方式查单。

### 7.6 后台商品管理

对应：

- `AdminProductService`
- `AdminProductServiceImpl`

能力包括：

- 查询商品列表
- 新增商品
- 修改商品
- 切换商品状态

关键点：

- 创建和更新时会做基础字段校验。
- SKU 需要唯一。

### 7.7 后台账号池管理

对应：

- `AdminAccountService`
- `AdminAccountServiceImpl`

能力包括：

- 查询账号池
- 导入账号
- 更新账号状态

关键点：

- 账号名、密码、备注都加密存储。
- 账号导入时做脱敏与摘要去重。
- 已经分配给订单的账号不能直接随便改状态。
- 每次变更后会回刷库存统计。

### 7.8 后台订单管理

对应：

- `AdminOrderService`
- `AdminOrderServiceImpl`

能力包括：

- 分页查询订单
- 查看订单详情
- 关闭订单

关闭订单时会做：

1. 锁定订单。
2. 校验订单当前必须是 `SUCCESS`。
3. 锁定该订单已分配的账号。
4. 把账号释放回账号池。
5. 更新订单为关闭状态。
6. 回刷商品库存和销量。

这保证了关闭订单不是简单改个状态，而是完整地回滚资源占用。

### 7.9 后台公告管理

对应：

- `NoticeService`
- `NoticeServiceImpl`

能力包括：

- 前台读取已发布公告
- 后台查看全部公告
- 新增公告
- 编辑公告
- 切换公告状态

## 8. 安全设计

这个项目在安全上已经做了几层基础保护。

### 8.1 Spring Security 权限隔离

配置文件：

- `SecurityConfig`

权限策略：

- 商品、公告、下单、注册、登录等接口公开访问。
- `/api/auth/me`、`/api/auth/orders` 需要会员身份。
- `/api/admin/**` 需要管理员身份。

### 8.2 JWT 无状态鉴权

核心类：

- `JwtService`
- `JwtAuthenticationFilter`
- `AuthenticatedUser`

做法：

- 登录成功后签发 JWT。
- Token 中写入 `uid`、`sub`、`scope`、`iat`、`exp`。
- 过滤器从 `Authorization: Bearer xxx` 中读取 token。
- 解析后把当前用户身份放入 Spring Security 上下文。

其中 `scope` 用于区分：

- `ADMIN`
- `MEMBER`

### 8.3 密码哈希

管理员和会员密码都不存明文，统一使用：

- `BCryptPasswordEncoder`

这是比较标准、可靠的密码存储方式。

### 8.4 敏感账号加密

核心类：

- `CryptoService`

处理内容：

- 账号名密文
- 账号密码密文
- 备注密文

加密算法：

- `AES/GCM/NoPadding`

同时保留：

- 脱敏字段供展示
- 摘要字段供去重

这意味着即使数据库被直接读取，也拿不到可直接使用的明文账号密码。

### 8.5 接口限流

核心类：

- `RateLimitFilter`

当前规则：

- `POST /api/orders`：1 分钟 10 次
- `POST /api/admin/auth/login`：15 分钟 5 次
- `POST /api/auth/login`、`POST /api/auth/register`：15 分钟 10 次

当前实现是内存级限流，适合单机开发和小规模部署。

### 8.6 全局异常与最小暴露

核心类：

- `GlobalExceptionHandler`

作用：

- 不直接把底层异常栈、SQL 细节返回给前端。
- 对校验异常、权限异常、业务异常做统一格式输出。

### 8.7 CORS 限制

通过 `ShopCorsProperties` 和 `SecurityConfig` 限制允许联调的来源地址，只开放给指定前端端口。

## 11. 当前仍可继续优化的点

虽然当前版本已经能运行，但从长期维护角度还有一些可以继续提升的地方。

### 11.1 配置安全性

目前数据库密码、JWT 密钥、AES 密钥直接写在 `application.yml`。

开发阶段这样没问题，但正式部署建议改成：

- 环境变量
- 配置中心
- Docker Secret
- K8s Secret

### 11.2 限流实现

当前 `RateLimitFilter` 是内存实现。

如果未来变成多实例部署，建议改成：

- Redis 限流
- 网关层限流

### 11.3 JWT 实现

当前 JWT 是自定义实现，轻量、可控，但正式生产可以考虑引入成熟 JWT 库，减少底层维护成本。

### 11.4 文本编码清理

当前项目里有一部分中文注释和初始化文案存在乱码表现，尤其在控制台输出时较明显。

建议后续统一做一次：

- 源文件 UTF-8 检查
- IDE 编码统一
- 数据初始化文案清理

### 11.5 测试覆盖

如果后续继续迭代，建议补更多自动化测试：

- service 层事务测试
- mapper SQL 测试
- 权限测试
- 限流测试
- 并发下单测试
